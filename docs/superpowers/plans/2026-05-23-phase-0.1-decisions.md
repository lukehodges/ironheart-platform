## D-03: First feature = org chart, not Tally replacement directly

**Original plan**: Seed 7 form templates from existing HTML questionnaires + ship portal audit tab. "Replaces Tally."

**Revised plan**: Build org chart FIRST. Forms are downstream of chart structure.

**Rationale**:
- Forms-only approach assumed headcount tiers drive routing (micro=1 form, mid=4 forms).
- Reality: micro company may need all 5 interviewed; 200-person company w/ 100 warehouse staff doesn't need 100 forms.
- Quote: "if we have something where it only allows us to speak to one, it doesn't work well... if 100 of them work in the warehouse, there's no point in speaking to 100 of those in the warehouse."
- Chart represents the actual interview plan. Forms get sent based on chart structure.

**Consequence**: Phase 0.1 ships chart only (no form sends). Phase 0.2 wires forms.

## D-04: Tier suggests defaults, never auto-applies

**Decision**: `resolveTier(teamSize)` runs on chart seed only. Sets default `interviewMode` per node. Consultant must click "Approve plan + send" before any form fires.

**Tier rules** (initial — tune in practice):

| Tier | Range | Default modes |
|---|---|---|
| MICRO | <5 | All nodes mode=ALL |
| SMALL | 5–15 | Functional heads ALL, ICs SAMPLE(2) |
| MID | 15–50 | Heads ALL, big depts SAMPLE(3), manual/floor depts SKIP (consultant judgement) |
| LARGE | 50+ | Owner + dept heads ALL, everything else SAMPLE(1) default |

**Rationale**:
- "Suggest only, consultant approves each" — chosen explicitly.
- Auto-fire risks over-sending forms to wrong people, damaging client relationship.
- Tier is starting point, not policy.

## D-05: Sampling = client owner picks names

**Decision**: When chart node has `mode=SAMPLE(N)`, the chart shows N empty slots under that dept. Client owner is tasked (in portal) with naming N people to fill those slots. No random selection.

**Rationale**:
- Quote: "they know who'll give honest answers."
- Owner judgment > algorithmic. Owner picks people who'll actually engage.
- Removes selection complexity from consultant side.

**Alternative rejected**: Random anonymous pulse to whole dept. Too noisy, loses owner steering, harder to follow up.

## D-06: Chart is collaborative, two-way, single source of truth

**Decision**: One `engagementOrgChart` table. Both consultant (Luke at `/platform/clients/[id]/onboarding`) and client owner (at `/[tenantSlug]/dashboard/onboarding`) read and write to the same rows. Activity log surfaces who-changed-what.

**Rationale**:
- Quote: "we, as consultants, build out as much as we can already, and then it gets passed off to the client, and they can edit it accordingly."
- Quote: "it should go back and forth where we can see the org chart, and they can see the org chart."
- Spreadsheet upload is an optional extra layer for clients who already have one.

**Mechanics**:
- Optimistic concurrency via `version` column per node.
- `lastEditedBy: "CONSULTANT" | "CLIENT"` on every row.
- Activity entries on every mutation. No realtime cursors / presence in 0.1.
- Some fields are consultant-only (e.g. `interviewMode`, `templateSlugOverride`) — client sees read-only.
- Some fields are anyone-writable (`label`, `headcount`, adding child nodes).

## D-09: WorkOS is the ONLY auth — no magic links

**Decision** (revised 2026-05-23): Tenant provisioning creates a real WorkOS Organization. Client owner is invited as a WorkOS user of that org. Login flow = WorkOS AuthKit for everyone (Luke + every client). Magic links are NOT used. `PortalSession` token mechanism is NOT used for auth (may still be used for one-off public links like booking, not for portal access).

**Rationale**:
- WorkOS AuthKit already integrated (see `docs/01-overview.md` — `@workos-inc/authkit-nextjs`).
- First 1M users free on WorkOS.
- Google SSO + MFA + future SAML come free.
- Magic links were a "minimum viable" shortcut. Building them now creates throwaway code + a second auth path to maintain.
- Cleaner mental model: one tenant = one WorkOS org. One user = one WorkOS identity. No bifurcation.

**Mechanics**:
- `provisioning.service.provisionClientTenant(engagementId)`:
  1. Create WorkOS Organization via `@workos-inc/node` SDK (`workos.organizations.createOrganization`)
  2. Store returned `workosOrgId` on `tenant` row
  3. Create internal `tenant`, `tenantModules`, `clientTenantId` on engagement
  4. Invite primary contact via `workos.userManagement.sendInvitation({ email, organizationId })`
  5. WorkOS sends the invite email (use WorkOS email template, override branding via WorkOS dashboard)
  6. Emit `tenant/provisioned` Inngest event
- Client clicks WorkOS invite link → completes WorkOS signup → lands at `/[tenantSlug]/dashboard`
- Our auth callback (`/auth/callback`) maps WorkOS user → internal `users` row → resolves their tenant via WorkOS org membership

**Drive folder**: Still stubbed in 0.1. Real Drive integration = Phase 4A/5.

**Consequence**:
- Phase 0.1.B drops magic-link handler, magic-link email, `PortalSession`-as-auth.
- Adds WorkOS SDK call wiring + invitation email branding setup (one-time, WorkOS dashboard).
- Client onboarding step ZERO is "click WorkOS invite" — no pre-portal magic-link landing.
- Slightly more setup time on first client (WorkOS dashboard config for invitation branding) but pays off forever.

## D-09.2: WorkOS role model

- **Primary contact at provisioning** = `admin` of their WorkOS org (manages own tenant — invites team, edits chart, approves work)
- **Bulk-imported / additional staff** = `member` by default
- **Admin promotion** = explicit action by existing tenant admin or platform admin (Luke)
- **Luke is NOT auto-member** of client tenants. "Add me to this tenant" platform action creates an `admin` invite for himself only when he wants to test.
- **App-level RBAC** via tRPC `requireTenantAdmin` middleware for: edit chart structure, invite users, accept deliverables, view billing. Members can: view chart, fill own form, view own bookings.
- **Invitation expiry** = 7 days (WorkOS default, no override)
- **Resend invitation** = built in 0.1 on engagement detail page (Task 9 in 0.1.B). Revokes existing + sends fresh invite w/ same role.

**Why admin for primary contact specifically:**
- They need to invite team members from their side (D-06 collaborative org chart)
- They edit company-side org chart, confirm details
- They approve audit plans, deliverables (later phases)
- Future: they manage billing, retainer renewals

**Why NOT default everyone to admin:**
- Bulk staff getting admin = security hole (warehouse staff shouldn't see invoices)
- When domain capture exists later (deferred), random `@northwind.com` signups shouldn't auto-admin
- WorkOS RBAC is per-org; defaulting members keeps least-privilege
