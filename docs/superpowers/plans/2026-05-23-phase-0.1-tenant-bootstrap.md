## Goal

Provision a real client tenant end-to-end via WorkOS. One tenant = one WorkOS Organization. Client owner gets invited via WorkOS, logs in via WorkOS, lands at `/[tenantSlug]/dashboard`. No magic links anywhere. No bifurcated auth.

---

## Scope

### In scope

- `/platform/clients/new` form (create customer + engagement)
- `provisioning.service.ts` — real implementation calling WorkOS Organizations + Invitations API
- WorkOS invitation email branding (one-time WorkOS dashboard config — Luke does this)
- WorkOS auth callback handler that maps invited user → internal `users` row → tenant scope
- Path-prefix tenant routing middleware: `/[tenantSlug]/*` resolves tenant context from WorkOS org membership
- `/[tenantSlug]/dashboard` landing page (minimal — "Welcome, [name]" + nav)
- **Tenant selector page `/select-tenant`** — when a WorkOS user belongs to multiple orgs (Luke testing as client of multiple, or any user invited to several tenants), render list of accessible tenants w/ logo + name + current engagement stage, click to enter. Also has "switch tenant" link in dashboard top-bar for in-session switching.

### Out of scope

- Google Drive folder (stubbed)
- Wildcard DNS / subdomain certs (path-prefix only — subdomain in later phase)
- WorkOS SAML / custom IDP (default Google SSO + email/password is enough)
- Full client portal feature set (just landing page in 0.1)
- Tenant deletion / suspension / lifecycle past creation
- Magic links — **entirely removed from plan**
- **Domain capture** — auto-joining users to a tenant based on email domain (e.g. `@northwind.com` → Northwind). Deferred to later phase. Risky to enable by default (generic domains like `@gmail.com`), needs per-tenant opt-in UI + admin verification flow.

## Architecture

```
Luke at /platform/clients/new
   ↓ form submit
consulting.service.createEngagement()
   ↓ creates customer + engagement (DISCOVERY)
   ↓
   [later] Luke transitions engagement → CONTRACTED
   ↓
Inngest event: engagement/stage-changed (toStage=CONTRACTED)
   ↓
provisioning.service.provisionClientTenant(engagementId)
   ├─ workos.organizations.createOrganization({ name, domains? })
   │     → returns workosOrgId
   ├─ create internal tenant row (slug + workosOrgId)
   ├─ enable client module set on tenantModules
   ├─ link engagement.clientTenantId = tenant.id
   ├─ workos.userManagement.sendInvitation({
   │       email: primaryContact.email,
   │       organizationId: workosOrgId,
   │       roleSlug: "admin"
   │     })
   │     → WorkOS sends branded invite email
   └─ emit event: tenant/provisioned
   ↓
Client receives WorkOS invite → clicks → completes WorkOS signup
(Google SSO or email/password — their choice)
   ↓
WorkOS redirects → /auth/callback?code=...
   ↓
src/app/auth/callback/route.ts
   ├─ exchange code for WorkOS session
   ├─ find or create internal users row (workosUserId)
   ├─ resolve tenant from user's WorkOS org memberships
   ├─ set authkit session cookie
   └─ redirect to /[tenantSlug]/dashboard
```

### Task 9: Invitation management UI

**Files**:

- New: `src/modules/consulting/provisioning.service.ts` — add `resendInvitation`, `inviteAdditionalUser`, `listPendingInvitations`, `revokeInvitation`

**UI** (on engagement detail page, right rail or new tab):
- "Tenant Access" panel
- List of users w/ WorkOS org membership:
  - Avatar + name + email + role pill (admin/member) + status (Active / Invited (pending) / Expired)
  - Actions per row: Resend invite (if pending/expired), Promote/Demote role (admin only mutation), Remove from org
- "Invite additional user" form:
  - Email, name (optional), role selector (default = `member`), note (optional)
  - Submit → calls `workos.userManagement.sendInvitation` w/ chosen role
- "Add me to this tenant" button (Luke-only) — sends invitation to Luke's email w/ `admin` role

**Service methods**:
- `resendInvitation({ tenantId, email })`:
  - Look up existing invitation in WorkOS by email + orgId
  - If pending → call `workos.userManagement.revokeInvitation` then `sendInvitation` again (same role)
  - If expired → call `sendInvitation` w/ same role as before
  - Return new invitation record
- `inviteAdditionalUser({ tenantId, email, role, name? })`:
  - Validate caller is admin of this WorkOS org (or platform admin)
  - Call `workos.userManagement.sendInvitation({ email, organizationId, roleSlug: role })`
  - Emit `tenant/invitation-sent` event
- `listPendingInvitations(tenantId)`:
  - Query WorkOS for all invitations on the org
  - Merge w/ our internal users (active) for unified UI
- `revokeInvitation({ tenantId, email })`:
  - Call `workos.userManagement.revokeInvitation`
  - Emit `tenant/invitation-revoked` event

**Permission gates**:
- `inviteAdditionalUser` w/ `role: "admin"` → only platform admin OR existing admin of that org can promote
- `inviteAdditionalUser` w/ `role: "member"` → any admin of that org
- `resendInvitation` → admin of org or platform admin
- `revokeInvitation` → admin of org or platform admin

Same UI panel also lives in **client-side** at `/[tenantSlug]/dashboard/settings/team` (Phase 0.2 likely, or stretch in 0.1) so client admin can invite their own team. For 0.1 just expose on platform side first.

## Acceptance

Creating a new client at `/platform/clients/new` + transitioning to CONTRACTED produces:
- Real WorkOS Organization
- Real WorkOS Invitation email sent
- Internal `tenant` row linked via `workosOrgId`
- Engagement `clientTenantId` set
- Client completes WorkOS signup → lands at correctly-scoped dashboard
- All ops idempotent (re-running creates nothing extra in WorkOS or DB)
- Zero magic links anywhere in the codebase

---

## Risks

- **WorkOS rate limits** — dev plan has limits. Verify quota for invitation sends. Production fine.
- **Slug collisions w/ WorkOS org names** — WorkOS uses its own org slug separate from our tenant slug. Don't try to keep them in sync — store `workosOrgId` only, generate our slug independently.
- **Invitation email branding** — by default WorkOS invitation says "You're invited to join {OrgName}". May need branded template config in WorkOS dashboard. Luke's one-time task before first real client.
- **Multi-org users** — Luke is in his own admin context + may end up as member of client orgs (if he wants to test as client). Auth callback's tenant resolution must handle "user belongs to multiple orgs" → tenant selector page. Edge case for 0.1; design now, build minimal selector.
- **Tenant scoping leaks** — middleware must attach `x-tenant-id` and all queries inside `/[tenantSlug]/*` must filter by it. Test cross-tenant access from tenant A's session against tenant B's data — must 403.
- **WorkOS Organization deletion** — if engagement gets deleted, WorkOS org orphans. Acceptable in 0.1, add cleanup in Phase 4A.
