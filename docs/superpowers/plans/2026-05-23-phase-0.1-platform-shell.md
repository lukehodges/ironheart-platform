## Goal

Give Luke a working `/platform/*` shell to land in. Sidebar nav, layout, route stubs. Not feature-complete — just a navigable skeleton that subsequent phases fill in.

## Why first

The org chart UI (0.1.C) and tenant create flow (0.1.B) both need a parent layout to mount inside. Build the shell once, everything else slots in.

---

## Scope

### In scope

- `src/app/platform/layout.tsx` — root layout w/ sidebar + top bar
- Sidebar component (reuse from `/admin/*` if compatible, or new at `src/components/platform/sidebar.tsx`)
- Top bar w/ search/notifications stubs
- Route stubs for core sections (empty pages with title + "Coming soon" placeholder + breadcrumb)
- Auth gate: redirect to `/auth/login` if no session OR `isPlatformAdmin` false

### Out of scope

- Real page content (each section is its own phase)
- Search functionality (placeholder only)
- Notifications inbox (placeholder)
- Migrating existing `/admin/*` pages

## Acceptance

- Visit `/platform` as Luke (platform admin) → lands on `/platform/today` shell
- Sidebar shows all sections, clicking each opens the stub page with breadcrumb
- Existing pages (`/platform/tenants`, `/platform/analytics`, etc.) still load and are wrapped by the new layout
- Visit `/platform` not logged in → redirects to login
- Visit `/platform` as non-admin user → redirects to `/[tenantSlug]/dashboard`

---

## Risks

- **Existing `/platform/*` pages may have their own layout** — check `src/app/platform/tenants/layout.tsx` etc. before adding a wrapper layout. Next.js layouts compose — adding a parent shouldn't break children, but verify.
- **Design token consistency** — `/admin/*` may use slightly different tokens than the DEMO_MAP design. Pick one source of truth before building sidebar. Use `.reference-design/index.html` if present.
- **Sidebar item count** — 25+ items at top level is too many. Group sensibly. May need a "More" collapse for less-used sections.
