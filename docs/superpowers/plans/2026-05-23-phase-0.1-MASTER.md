## Why this exists

Big consulting-pipeline plan exists (Phases 1A–7 in `2026-05-10-consulting-pipeline-MASTER.md`) but most of it is unbuilt past Phase 1A. Reality check 2026-05-23:

- `/admin/*` has 21 working pages (legacy migration target)
- `/platform/*` has only 6 superadmin pages (tenants/products/subscriptions/revenue/analytics)
- `/dashboard/*` has 6 client portal pages
- No tenant has been provisioned end-to-end. Luke is still working out of `/admin/*`.

Phase 0.1 is a **vertical slice** to prove the platform↔subtenant flow works before committing to the bigger phases. It's the smallest thing that:

1. Gives Luke a working `/platform/*` shell (his world)
2. Provisions one real client tenant
3. Builds the org chart feature end-to-end so we can see consultant→client collaboration working

If 0.1 works, Phase 4A (heavy WorkOS provisioning) becomes incremental. If 0.1 reveals problems, we catch them before building the rest.

## Architecture decisions (locked)

From conversation 2026-05-23 (see `2026-05-23-phase-0.1-decisions.md` for full rationale):

| Decision | Choice |
|---|---|
| Tier model | **Luke flat in `/platform/*`**. No Ironheart-tenant split. Defer until staff hired. |
| Client URL | **`slug.ironheart.app/dashboard`** (subdomain, eventual). Phase 0.1 ships path-prefix `ironheart.app/[tenantSlug]/dashboard` as cheap fallback. |
| First user-facing feature | **Org chart-driven onboarding**, not Tally form replacement directly. Forms are downstream of chart. |
| Routing logic | Org chart drives form sends, NOT headcount tiers. Tier suggests defaults only. |
| Sampling | Client owner picks names for SAMPLE(N) nodes. No random. |
| Chart edits | Collaborative two-way. Consultant + client write to same table. Activity log surfaces diffs. |

## Scope (what 0.1 ships)

### In scope

1. **Platform shell skeleton** — `/platform` layout, sidebar nav, route stubs for the 20-ish core pages. Doesn't need to be feature-complete; needs to be a place to land.
2. **Tenant bootstrap (WorkOS-only)** — create-tenant flow in `/platform/clients/new`, real WorkOS Organization creation, WorkOS invitation to client owner, WorkOS auth callback resolves them to `/[tenantSlug]/dashboard`. **No magic links** — WorkOS is the only auth.
3. **Org chart schema + service** — `engagementOrgChart` table, repo, service w/ `resolveTier`, `seedChartFromTier`, `planOnboardingForms`, `enqueueOnboardingForms`, `getOnboardingStatus`.
4. **Org chart UI — consultant side** — `/platform/clients/[id]/onboarding` editor. Tree, drag-to-reparent, per-node mode selector, template override dropdown, activity log.
5. **Org chart UI — client side** — `/[tenantSlug]/dashboard/onboarding`. Same chart, client-edit perms, read-only fields for consultant-only attributes.
6. **Path-prefix tenant routing** — Next middleware resolves `/[tenantSlug]/*` → tenant context. Wildcard subdomain DNS deferred.

### Out of scope (for 0.1)

- Form template seeds (Phase 0.2)
- Form sending engine wiring (Phase 0.2 — `planOnboardingForms` exists but `enqueueOnboardingForms` no-ops)
- Google Drive folder creation (stubbed)
- Subdomain DNS / wildcard certs
- Migrating `/admin/*` → `/platform/*` (legacy stays parallel)
- Audit workspace UI changes
- Report generator
- Activity feed cross-module unification (chart activity is local to chart for 0.1)

## Success criteria

When 0.1 is done, this sequence works end-to-end:

1. Luke logs in at `ironheart.app/platform` → sees shell w/ sidebar
2. Goes to `/platform/clients/new` → creates customer + engagement at DISCOVERY
3. Transitions engagement → CONTRACTED → triggers tenant provisioning (real WorkOS org + invitation email + tenant row + chart seed)
4. Goes to `/platform/clients/[id]/onboarding` → org chart auto-seeded from `qualificationData.teamSize` w/ tier defaults
5. Luke edits chart: marks warehouse SKIP, ops dept SAMPLE(3), adds 4 named PERSON nodes
6. Client receives WorkOS invitation email (sent at step 3 via WorkOS API)
7. Client clicks WorkOS invite → completes WorkOS signup (Google SSO or email/password) → lands at `ironheart.app/[tenantSlug]/dashboard/onboarding`
8. Client sees same chart, adds 3 names to ops SAMPLE slots, edits 1 role label
9. Luke refreshes `/platform/clients/[id]/onboarding` → sees client's edits w/ activity entries ("Sarah Chen named for Ops sample slot")
10. Luke clicks "Approve plan" — no forms send yet (Phase 0.2), but plan preview shows N forms across X recipients

If steps 1–10 work without manual SQL, 0.1 is done.
